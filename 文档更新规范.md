## 一、问题模式总结（基于任务001）

### 1.1 常见错误类型
根据《文档差异分析报告》，识别出以下问题模式：

| 问题类型 | 问题描述 | 影响等级 | 预防措施 |
|---------|---------|---------|---------|
| **拼写错误** | 类名、方法名拼写错误（如 `EventLinsener` → `EventListener`） | 高 | 使用代码引用规范，自动验证 |
| **模块名称过时** | 引用已废弃或更名的模块（如 `framework-ai-jmanus`） | 高 | 定期对照 pom.xml 验证模块名 |
| **版本信息过时** | 版本号未更新（如 4.1 → 4.2.0） | 中 | 建立版本检查清单 |
| **API变更** | 注解、接口参数变化未同步更新 | 高 | 代码对比验证 |
| **内容不完整** | 缺少新功能说明或示例代码 | 中 | 使用完整性检查清单 |

### 1.2 最佳实践原则
- **KISS原则**：保持简洁，避免过度冗余
- **DRY原则**：重复内容提取到公共文档
- **YAGNI原则**：只记录必要的内容，避免过度设计
- **单一职责**：每个文档聚焦一个主题

## 二、代码引用格式规范

### 2.1 类名、接口名、注解名引用

#### 规范要求
```markdown
- 类名：使用代码格式，首字母大写
  示例：`BaseDao`、`StudentEntity`

- 接口名：使用代码格式，以 I 开头
  示例：`IStudentDao`、`ICache`

- 注解名：使用代码格式，以 @ 开头
  示例：`@Dao`、`@Cache`、`@Transactional`

- 方法名：使用代码格式，首字母小写
  示例：`getById()`、`save()`、`queryPager()`
```

#### 完整包名引用（首次出现时）
```markdown
首次引用时需包含完整包名：
`com.hbasesoft.framework.db.core.BaseDao`

后续可直接使用简写：`BaseDao`
```

### 2.2 代码文件引用格式

#### 源代码文件引用
```markdown
> 文件路径（相对于项目根目录）

示例：
> framework-db/framework-db-core/src/main/java/com/hbasesoft/framework/db/core/BaseDao.java
```

#### 配置文件引用
```markdown
> 配置文件路径

示例：
> src/main/resources/application.yml
```

### 2.3 方法签名展示规范

#### 标准格式
```java
/**
 * 方法功能简述
 *
 * @param paramName 参数说明
 * @return 返回值说明
 */
ReturnType methodName(ParamType paramName);
```

#### 示例
```java
/**
 * 根据ID查询单个实体
 *
 * @param id 实体ID
 * @return 实体对象，不存在则返回null
 */
T get(Serializable id);
```

### 2.4 代码示例格式规范

#### 基本要求
1. **完整性**：代码必须可运行，包含必要的导入语句
2. **上下文**：提供足够的上下文，避免片段化
3. **注释**：关键逻辑添加中文注释
4. **缩进**：使用4空格缩进

#### 标准模板
```java
// 导入语句（完整且必要）
import com.hbasesoft.framework.db.core.BaseDao;
import org.springframework.stereotype.Service;
import javax.annotation.Resource;

/**
 * 类功能说明
 */
@Service
public class ExampleService {

    @Resource
    private IStudentDao studentDao;

    /**
     * 方法功能说明
     */
    public void exampleMethod() {
        // 实现逻辑
        // 添加关键注释
    }
}
```

### 3.1 可运行性要求

- [ ] 类定义完整（包含类名、访问修饰符）

#### 验证方法
```bash

javac -classpath "..." Example.java
```

### 3.2 注释规范

#### 类注释
```java
/**
 * 类功能简述
 *
 * <p>详细说明（可选）</p>
 *
 * @author 作者名
 * @since 版本号
 */
```

#### 方法注释
```java
/**
 * 方法功能简述
 *
 * @param paramName 参数说明
 * @return 返回值说明
 * @throws Exception 异常说明（如适用）
 */
```

#### 行内注释
```java
// 关键步骤说明
业务逻辑代码;
```

### 3.3 命名规范

#### 实体类命名
- **PO类**：以 `Entity` 或 `PO` 结尾
  - 示例：`StudentEntity`、`UserPO`
- **DAO接口**：以 `IDao` 或 `Dao` 结尾
  - 示例：`IStudentDao`、`UserDao`
- **Service类**：以 `Service` 结尾
  - 示例：`StudentService`、`UserService`

#### 变量命名
- **常量**：全大写，下划线分隔
  - 示例：`MAX_SIZE`、`DEFAULT_TIMEOUT`
- **成员变量**：驼峰命名
  - 示例：`studentDao`、`cacheHelper`
- **局部变量**：驼峰命名，见名知意
  - 示例：`userName`、`pageIndex`

## 六、质量验证标准

### 6.1 评分体系

#### 总分计算公式
```
总分 = 准确性得分 × 40% + 完整性得分 × 30% + 可读性得分 × 20% + 实用性得分 × 10%
```

#### 评级标准
- **优秀**：总分 ≥ 90分
- **良好**：80分 ≤ 总分 < 90分
- **合格**：70分 ≤ 总分 < 80分
- **不合格**：总分 < 70分

### 6.2 准确性标准（权重40%）

#### 评分项（每项10分，共100分）
| 评分项 | 评分标准 |
|-------|---------|
| 类名准确性 | 所有类名与代码完全一致（10分）；每错1个扣2分 |
| 方法签名准确性 | 所有方法签名参数、返回值正确（10分）；每错1个扣2分 |
| 包路径准确性 | 所有包路径准确（10分）；每错1个扣2分 |
| 参数说明准确性 | 所有参数类型、说明正确（10分）；每错1个扣1分 |
| 代码可运行性 | 所有示例代码可运行（20分）；每处错误扣5分 |
| 配置参数准确性 | 所有配置参数名、默认值正确（10分）；每错1个扣2分 |
| 版本信息准确性 | 版本号、变更历史准确（10分）；信息过期扣5分 |
| API变更同步 | API变更已同步到文档（10分）；遗漏1处扣3分 |
| 异常说明准确性 | 异常类型、触发条件准确（5分）；每错1个扣2分 |
| 返回值说明准确性 | 返回值类型、说明准确（5分）；每错1个扣2分 |

#### 验证方法
```bash
# 1. 编译验证示例代码
mvn compile -Dincludes="**/test/**/*Test.java"

# 2. 对照源代码验证
find . -name "*.java" -path "*/framework-db/*" | xargs grep "class BaseDao"

# 3. 运行单元测试
mvn test -Dtest=BaseDaoTest
```

## 七、文档更新工作流程

### 7.1 更新触发条件

#### 必须更新文档的情况
1. **API变更**：添加、修改、删除公开API
2. **配置变更**：新增、修改、删除配置项
3. **版本发布**：框架版本号更新
4. **功能新增**：新增重要功能
5. **功能废弃**：标记API或功能为废弃
6. **Bug修复**：修复影响使用的Bug
7. **用户反馈**：收到3个以上相同问题的反馈

#### 建议更新文档的情况
1. 代码优化导致用法变化
2. 发现文档错误或不准确
3. 示例代码可以改进
4. 文档结构可以优化

### 7.2 更新流程

#### Step 1：识别变更
```bash
# 查看Git变更记录
git diff HEAD~1 HEAD -- "*.java" | grep "public\|protected"

# 查看API变更
git log --oneline --all --grep="API" | head -20
```

#### Step 2：对照代码验证
```bash
# 1. 找到对应的Java文件
find . -name "BaseDao.java"

# 2. 验证方法签名
grep -A 5 "public.*get(" BaseDao.java

# 3. 验证注解
grep -B 2 -A 10 "@Dao" IStudentDao.java
```

#### Step 3：更新文档
1. 按照本文档规范更新内容
2. 使用完整性检查清单验证
3. 使用质量验证标准评分

#### Step 4：验证示例代码
```bash
# 编译验证
mvn clean compile

# 运行测试
mvn test -Dtest=*Test

# 代码格式化
mvn fmt:format
```

#### Step 5：提交审核
```bash
# 提交变更
git add docs/*.md
git commit -m "docs: 更新XX模块文档，新增XX功能说明"

# 推送到远程仓库
git push origin branch-name
```

### 9.1 如何确保代码示例可运行？

#### 问题
示例代码缺少必要的导入语句或上下文，无法直接运行。

#### 解决方案
1. 完整导入语句
```java
// ❌ 错误示例
studentDao.get(id);

// ✅ 正确示例
import com.hbasesoft.framework.db.core.BaseDao;
import org.springframework.stereotype.Service;
import javax.annotation.Resource;

@Service
public class StudentService {
    @Resource
    private IStudentDao studentDao;

    public StudentEntity getStudentById(String id) {
        return studentDao.get(id);
    }
}
```

2. 添加完整上下文
3. 在IDE中验证可编译
4. 添加注释说明依赖

### 9.2 如何处理API版本兼容性？

#### 问题
新版本废弃了旧API，如何标注迁移路径？

#### 解决方案
```markdown
#### ⚠️ 已废弃（4.2.0）

**原API**：`query(String sql, Class<?> clazz)`
**替代方案**：`query(QueryWrapper<?> wrapper)`
**废弃版本**：4.2.0
**移除版本**：5.0.0

**废弃原因**：新API提供类型安全和更好的可读性

**迁移示例**：
```java
// 旧方式（已废弃）
List<Student> list = studentDao.query("SELECT * FROM t_student", Student.class);

// 新方式（推荐）
List<Student> list = studentDao.query(q -> q.eq("deleted", false));
```
```

### 9.3 如何处理配置参数变更？

#### 问题
配置参数名称或默认值发生变化。

#### 解决方案
```markdown
#### 配置变更（4.2.0）

| 旧参数名 | 新参数名 | 默认值 | 说明 |
|---------|---------|-------|------|
| cache.timeout | cache.expireTime | 600 | 缓存过期时间（秒） |

**迁移指南**：
在 `application.yml` 中将 `cache.timeout` 替换为 `cache.expireTime`
```

### 9.4 如何避免拼写错误？

#### 问题
类名、方法名拼写错误（如 `EventLinsener`）。

#### 解决方案
1. **使用代码引用**
```markdown
# ❌ 错误：直接手动输入
EventLinsener

# ✅ 正确：从代码中复制
EventListener
```

2. **使用IDE自动补全**
   - 在IDE中输入类名，使用自动补全
   - 复制粘贴到文档

3. **使用脚本验证**
```bash
# 验证类名是否存在
find . -name "EventListener.java"

# 验证包路径
grep -r "class EventListener" framework-*
```
